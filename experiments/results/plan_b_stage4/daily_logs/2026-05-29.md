- Do not use partial `D_naive_union` or `A_raw` runs as final downstream evidence; only runs with `status=trained`, `steps>=2000`, and adapters are eligible for VQAv2 eval.
- Keep the resumed queue running rather than starting ad hoc one-off commands, so completed splits are skipped consistently and missing splits remain visible in the queue log.
- Treat strict VQAv2 exact/consensus as a short-answer metric, not the only downstream signal. For caption-style LoRA adapters, report relaxed answer-containment metrics alongside strict scores and clearly label them as auxiliary.
- Treat current 25K downstream runs as a first v2 diagnostic unless the sampling policy is repaired: `run_llava_stage4_lora.py` currently slices the first `max_samples` JSON records before DataLoader shuffling, so split-level differences can reflect manifest order / prefix composition rather than only the dedup policy. A paper-facing rerun should random-sample the 25K budget from each split with a fixed seed before training.
- To test whether Stage 4 E can match or beat A/B/D under a fairer VQA readout, run a short-answer prompt eval (`Answer with a short phrase.`, `max_new_tokens=6`), include a no-adapter LLaVA baseline, and sweep E checkpoints before making final downstream claims.
- If using `E` checkpoint-step-1500 as the reported downstream model, apply the same early-checkpoint selection rule to A/B/C/D or report both final-checkpoint and best-checkpoint rows. Do not give E an unfair oracle checkpoint advantage.
- Downstream checkpoint-selection policy: for the paper-facing best-checkpoint comparison, select the best checkpoint for every group from the same sweep set (`0500`, `1000`, `1500`, `1900`, `2000`) using strict `vqa_consensus` as the primary metric because it is closer to the official VQA accuracy protocol. Report `relaxed_vqa_consensus` only as an auxiliary diagnostic for caption-style long-answer undercounting. Keep final-checkpoint results as an overtraining / forgetting diagnostic.
